本文件是 Claude / 其他执行代理接手 `game-video-tool` 前必须先读的总入口。它的优先级高于临场判断：任何开发、排障、上线、回滚都必须先满足这里的硬规则，再继续执行。

- 任何修复都要优先保证多人并发、稳定性、响应速度、可回滚和长期维护。

## 3. 已完成进度

### 3.1 生产备份、风险分级与回滚基础

已完成：

- 修复并使用 `deploy/backup-game-video-data.sh`
- 支持生产数据目录快照
- 支持 SQLite `.backup`
- 支持备份保留数量配置
- 已配置服务器定时备份 cron
- 已实际生成过生产备份

仍要坚持：

- R0/R1 不默认做生产数据全量备份，避免无意义占满磁盘。
- R2 优先做范围备份；范围不清时升级为 R3。
- R3 才默认执行生产数据全量备份，且必须先检查磁盘空间。
- 任何级别都必须确认发布包不会覆盖 `/home/deploy/game-video-data`。

### 3.1.1 视频工作台架构治理现状

已完成：

- `GameVideoPage.jsx` 已拆出多个受控面板：生成、替换、图片、反推、设置、生成记录等。
- 已拆出多个 hook：任务轮询、自动保存、项目加载、项目动作、bootstrap、tabState、tab 持久化、媒体资源、文字插入。
- 后端已具备模型 registry、视频参数校验、任务状态查询限流/去重、provider queue、健康报告、任务审计和磁盘生命周期治理。

仍要坚持：

- `GameVideoPage.jsx` 仍是 orchestrator，不能继续塞新大功能。
- `components/` 只做展示和事件转发，禁止直接调用 API、缓存、轮询、保存。
- `use*.js` hook 必须有明确职责，不能机械搬代码造成新的散乱。
- `game_routes.py` 仍是生产关键大路由，暂不和前端拆分混在一个包里处理。
- 详细边界见 `docs/WORKBENCH_ARCHITECTURE_GOVERNANCE.md`。

### 3.2 云同步安全

已完成：

- `server/cloud_sync.py` 已增加自同步保护，避免 `cloud_url == PUBLIC_BASE_URL` 时出现自己同步自己、请求放大或历史文件堆积的问题。

仍需关注：

- 历史 `cloud-dbs` 文件较多，后续需要单独做清理方案，不要在功能发布中顺手删除。

### 3.3 提示词优化与参考媒体分析

已完成：

- `analyze_prompt` / `refresh_prompt` 支持传入并读取参考视频、进阶参考视频、角色图、场景图。
- Gemini 链路会发送真实图片/视频 bytes。
- GPT / OpenAI 链路会从视频中抽帧，再通过视觉输入分析。
- GPT 被限流或失败时不会自动退回 Gemini；会按 GPT 流程重试，重试后仍失败才给用户明确原因。
- 提示词优化默认模型已调整到更稳的 `gemini-2.5-flash`。
- 后端 keepalive 已处理 `CancelledError`，减少用户中断导致的异常噪声。

仍需重点排查：

- 同事反馈“提示词优化显示延迟然后失败”，要继续按四个模型逐个验证真实可用性和错误提示。
- GPT 网页端可直接接视频，但当前 API 实现是视频抽帧后走视觉输入；不要声称等同网页端完整视频理解能力，除非已经用官方文档和实测确认。

### 3.4 图片生成比例选择

## 6. 每次任务开始前必须做

1. 读取本文件。
2. 读取相关细则：
   - `docs/CURRENT_STATUS_AND_ROADMAP.md`
   - `docs/TESTING_RELEASE_AND_BACKUP_POLICY.md`
   - `docs/RELEASE_ROLLBACK_AND_BACKUP_SOP.md`
   - `docs/CHANGE_CHECKLIST.md`
3. 执行 `git status --short`。
4. 判断本地是否有无关脏改。
5. 明确本次任务只允许修改哪些文件。
6. 如果发现未完成 WIP，不得默认发布。

## 7. 开发流程

每次开发按以下顺序：

1. 定义问题
   - 用户遇到什么问题？
   - 影响哪个功能？
   - 是线上问题、体验问题、性能问题，还是未来功能？

2. 确认范围
   - 改前端、后端、部署、数据库还是第三方 API？
   - 是否影响登录、权限、文件、同步、任务状态、计费成本？

3. 最小修改
   - 优先沿用现有结构。
   - 不做无关重构。
   - 不顺手发布未完成功能。

4. 本地验证
   - 前端：`npm run build`
   - 后端：至少 `py_compile`，能启动则做 `/health`
   - 相关接口：必须最少调用一次

5. 汇报结果
   - 改了什么
   - 测了什么
   - 没测什么
   - 风险是什么

6. 等用户确认
   - 没有用户明确同意，不推送，不部署。

## 8. 上线硬流程

任何上线必须逐项通过：

1. `git status --short`
   - 确认本次发布文件范围。
   - 发现无关 WIP 必须停下说明。

2. `git diff`
   - 人工审查是否混入未完成入口、调试代码、密钥、临时代码。

3. 构建和编译
   - 前端改动：`cd react-ui && npm run build`
   - 后端改动：`.venv/bin/python -m py_compile ...`

4. 关键词扫描
   - 搜索未发布功能关键词。
   - 例如买量分析未发布时，必须确认生产 bundle 不包含：
     - `买量分析`
     - `creative-analysis`
     - `/api/creative`
     - `CreativeAnalysis`

5. 备份
   - 必须先完成 R0/R1/R2/R3 备份风险分级。
   - R0 文档包不部署生产，不需要生产 DB 备份。
   - R1 代码包保留代码回滚点和发布包，不做全量数据备份。
   - R2 做相关 DB 或目录范围备份。
   - R3 才执行生产数据全量备份，并记录备份路径、确认文件非空。

6. 部署
   - 只上传本次允许发布的代码和构建产物。
   - 不上传生产数据目录。
   - 不把 `.claude/worktrees`、临时文件、未完成 WIP 当作发布内容。

7. 重启
   - 优先使用服务管理方式。
   - 如果只能 kill 端口进程并依赖 systemd 自动拉起，必须提前告知可能有短暂不可用窗口。

8. 线上验收
   - `/health` 返回 200
   - `/api/version` 返回预期版本
   - 首页返回 200
   - 新 JS/CSS 返回 200
   - 本次功能实际可用
   - 未发布功能不可见

9. 发布记录
   - 记录版本号、备份路径、发布文件、测试结果、风险和回滚方式。

## 9. 回滚规则

触发回滚：

- `/health` 失败
- 首页打不开
- 登录异常
- 关键功能不可用
- 大量用户反馈无法连接
- 延迟明显升高且短时间不能定位
- 未完成功能误上线

回滚顺序：

1. 停止继续发布新改动。
2. 记录当前错误。
3. 优先代码回滚到上一稳定版本。
4. 验证 `/health`、首页、关键功能。
5. 只有确认数据损坏时，才考虑数据恢复。

禁止：

- 未确认原因就恢复数据。
- 直接删除生产文件。
- 用新补丁覆盖事故现场且不留记录。

## 10. 多人并发与稳定性要求

任何会被多人同时使用的功能，都必须考虑：

- 请求超时是否合理。
- 上游 AI 慢或失败时是否有明确提示。
- 重试是否有限制，不能无限循环。
- 任务状态是否能落库。
- 任务失败是否能回写。
- 轮询是否会停止。
- 并发上传是否互相污染。
- 切项目时是否会把旧项目状态写到新项目。
- 长任务是否用后台任务或可轮询任务承载。
- 用户级数据是否隔离。

上线前至少做：

- 多次并发健康检查。
- 本次相关接口的并发或连续请求测试。
- 查看最近日志是否新增 500、导入错误、数据库锁、超时堆积。

## 11. 性能与成本要求

长期维护目标：

- 响应迅速：普通读取接口应尽量保持秒级以内。
- 稳定优先：AI 上游慢时不能拖垮整个服务。
- 成本可控：不要无意义重复调用昂贵模型。
- 可持续优惠：优先复用已有 API Key 配置、模型选择和用户级 Key 体系，不硬编码高成本路径。
- 可观测：失败要记录 provider、model、任务 ID、错误摘要。

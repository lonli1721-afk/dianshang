# 发布、回滚与备份 SOP

本文档是本项目的标准操作流程（SOP）。目标是让发布、回滚、备份和验收都有固定动作，不依赖临场记忆。

## 1. 适用范围

适用于以下场景：

1. 本地需求开发完成后的发布。
2. 线上故障后的回滚。
3. 重要改动前后的备份操作。
4. 涉及认证、权限、同步、数据路径、部署脚本的高风险上线。

## 2. 角色理解

在当前项目协作中，默认角色如下：

1. 用户：决定是否推送、是否部署、是否接受风险。
2. Codex/AI：负责分析、改动、测试、整理结果、执行经确认的推送/部署。

原则：

1. Codex 可以默认自动提交。
2. 推送前必须征求用户确认。
3. 部署前必须征求用户确认。

## 3. 发布前 SOP

### 3.1 确认改动范围

发布前先明确：

1. 改了哪些文件。
2. 涉及前端还是后端。
3. 是否影响登录、权限、同步、备份、数据路径。
4. 是否需要运行构建。
5. 是否需要执行额外备份。

### 3.2 运行测试

按改动类型执行测试：

1. 前端改动：
   - `npm run build`
   - 如可行，`npm run lint`
   - 打开关键页面验证
2. 后端改动：
   - 启动服务
   - 访问 `/health`
   - 验证关键接口
3. 高风险改动：
   - 额外验证登录、权限、同步、备份、错误路径

### 3.3 汇总测试结果

发布前必须形成一份简明结论：

1. 哪些测试通过。
2. 哪些未执行。
3. 当前风险是什么。
4. 是否建议进入推送。
5. 是否建议进入部署。

### 3.4 生成代码回滚点

发布前必须至少满足以下一项：

1. 已有本地 Git 提交。
2. 已有远程仓库提交。
3. 已有带时间戳的发布包。

推荐同时具备：

1. 本地提交。
2. 远程推送。
3. 发布包归档。

### 3.5 备份确认

部署前必须先做备份风险分级，再执行对应备份动作：

| 级别 | 典型范围 | 部署前动作 |
| --- | --- | --- |
| R0 | 文档、提示词、交接材料、注释说明 | 不做生产数据备份，记录变更即可 |
| R1 | 纯前端、样式、只读接口、无数据写入代码 | 代码提交、发布包、回滚点；不做全量数据备份 |
| R2 | 单功能写入、文件输出、费用记录、任务状态、模型结果缓存 | 备份相关数据库或相关目录；范围不清时升级 R3 |
| R3 | 认证、权限、账号、数据库结构、数据迁移、同步、备份/部署脚本、数据目录路径、大版本上线 | 执行生产数据全量备份 |

部署前需要确认：

1. 已记录本次备份风险级别。
2. 数据目录不在覆盖范围内。
3. 对应级别的备份策略已经执行或明确不需要执行。
4. 若执行备份，备份产物路径明确。
5. 如果无法判断风险级别，按更高一级执行。

## 4. 推送 SOP

### 4.1 推送前

执行顺序：

1. 确认工作区状态清晰。
2. 确认当前分支。
3. 确认最新提交信息。
4. 将测试结果告知用户。
5. 获取用户确认。

### 4.2 推送执行

标准动作：

1. 推送当前分支。
2. 确认远程跟踪关系正常。
3. 记录推送后的提交号。

### 4.3 推送后

推送完成后告知用户：

1. 推送到哪个仓库。
2. 推送了哪个分支。
3. 最新提交号是什么。
4. 是否建议继续部署。

## 5. 部署 SOP

### 5.1 部署前确认

部署前必须逐项确认：

1. 测试通过。
2. 用户明确同意部署。
3. 已确认代码版本。
4. 已确认数据目录不可覆盖。
5. 已确认备份要求满足。
6. 已准备健康检查与首页验收命令。

### 5.2 本项目标准部署思路

当前长期目标是使用近零停机发布。新包发布优先参考
[`docs/ZERO_DOWNTIME_RELEASE.md`](/Users/jinyu/Documents/game-video-tool/docs/ZERO_DOWNTIME_RELEASE.md)：

1. 新版本先在备用端口启动。
2. 备用端口通过 `/health`、首页、关键接口 smoke。
3. nginx 切流到备用端口。
4. 再停止旧端口实例。

如果近零停机基础设施尚未启用，才允许使用旧的单实例部署方式。旧方式有明确停机窗口，必须在发布记录中标注风险，并尽量选择低峰期执行。

根据当前项目交接文档，旧的单实例部署思路是：

1. 本地代码修改完成。
2. 如涉及前端，先构建前端。
3. 打包项目代码。
4. 上传到服务器。
5. 服务器解压覆盖代码目录。
6. 重启服务。
7. 执行健康检查和首页验收。

核心原则：

1. 只覆盖代码。
2. 不覆盖数据目录。
3. 前端发布不得直接删除所有旧 `assets/*.js` / `assets/*.css`。必须保留最近一批旧 hashed chunk，避免已经打开旧页面的用户切换路由时请求旧 chunk 404 白屏。
4. 前端 `dist` 替换优先使用 `deploy/install-frontend-dist.sh`，该脚本会先备份当前 `dist`，再把旧 chunk 合并进新 `dist` 后切换。

前端 `dist` 标准安装示例：

```bash
APP_DIR=/home/deploy/game-video-tool \
BACKUP_DIR=/home/deploy/game-video-backups \
RELEASE_ID=frontend-r1-$(date +%Y%m%d-%H%M%S) \
deploy/install-frontend-dist.sh /tmp/release/game-video-tool/react-ui/dist
```

如果发布后仍看到旧 chunk 404，先补兼容资源或保留旧资源，再观察新日志窗口；不要直接回滚业务代码，除非首页、登录或核心功能不可用。

### 5.3 部署后验收

最低验收动作：

1. 检查 `/health` 返回 200。
2. 检查首页可访问。
3. 检查前端静态资源被正确加载。
4. 检查本次改动的关键功能至少一次。
5. 检查新旧前端 chunk 可访问：当前 `index.html` 引用的主 chunk 必须 200，上一版保留 chunk 不应 404。

建议在部署前后各执行一次只读巡检：

1. 在服务器代码目录运行 `deploy/production-inspect.sh`。
2. 如生产路径不同，通过环境变量覆盖，例如：
   `APP_DIR=/home/deploy/game-video-tool DATA_DIR=/home/deploy/game-video-data BACKUP_DIR=/home/deploy/game-video-backups LOG_FILE=/home/deploy/game-video-tool/app.log deploy/production-inspect.sh`
3. 对比巡检输出中的磁盘、目录体积、日志错误/429/5xx 计数、Gemini key 数量尾号、前端 dist 文件和当前进程。
4. 巡检脚本只读，不删除、不重启、不修改配置；异常项应先记录，再决定是否继续部署或回滚。

上线后任务状态观察优先使用只读脚本：

```bash
cd /home/deploy/game-video-tool
deploy/task-stale-watch.py \
  --data-dir /home/deploy/game-video-data \
  --stale-seconds 300 \
  --critical-count 1 \
  --report-dir /home/deploy/game-video-backups/task-stale-watch
```

该脚本只读用户数据库并写入小体积 JSON/TXT 报告；它不修复任务、不调用上游 provider、不重启服务、不写业务数据。若发现 stale processing，先保存报告，再执行只读 probe；任何 repair 都必须重新获得用户明确批准。

失败原因观察使用只读日报：

```bash
cd /home/deploy/game-video-tool
deploy/task-failure-report.py \
  --data-dir /home/deploy/game-video-data \
  --since-hours 24 \
  --report-dir /home/deploy/game-video-backups/task-failure-report
```

该报告按 provider、model、user、error category 聚合最近失败任务，并输出已脱敏的错误样本。报告只用于定位方向；不能根据报告自动降级模型、自动重试、自动 repair 或写线上数据。

### 5.4 部署失败处理

如果部署失败：

1. 先停止继续变更。
2. 保留错误信息。
3. 判断是构建问题、启动问题、代理问题还是代码问题。
4. 若无法快速修复，进入代码回滚流程。

## 6. 回滚 SOP

### 6.1 回滚触发条件

出现以下任一情况，优先考虑回滚代码：

1. 健康检查失败。
2. 首页打不开。
3. 关键功能不可用。
4. 登录异常。
5. CPU、内存、请求延迟出现明显异常且短时间无法修复。

### 6.2 回滚原则

1. 优先回滚代码，不直接碰数据。
2. 先恢复服务可用性，再做根因分析。
3. 数据恢复是最后手段，不是默认操作。

### 6.3 代码回滚动作

可选回滚来源：

1. 上一个稳定 Git 提交。
2. 上一个稳定发布包。

推荐动作：

1. 选定回滚版本。
2. 重新部署该版本代码。
3. 重启服务。
4. 执行健康检查和首页验收。

### 6.4 数据恢复动作

只有在以下条件同时成立时才考虑：

1. 已确认不是单纯代码问题。
2. 已确认数据损坏或数据结构异常。
3. 已存在可用备份。
4. 已明确恢复范围和影响。

禁止事项：

1. 未确认问题性质前直接恢复数据。
2. 用数据恢复替代代码回滚。

## 7. 备份 SOP

### 7.1 代码备份 SOP

标准动作：

1. 改动完成后形成 Git 提交。
2. 需要异地保存时推送到 GitHub。
3. 重要上线前保留一个可识别版本点。

### 7.2 本地自动备份 SOP

当前项目已经具备本地数据库自动备份能力。

核对动作：

1. 确认应用运行目录下存在备份目录。
2. 确认备份目录持续生成新内容。
3. 如改动数据库逻辑，观察备份行为是否仍正常。

### 7.3 服务器数据备份 SOP

R3 高风险部署前必须执行：

1. 运行 [`deploy/backup-game-video-data.sh`](/Users/jinyu/Documents/game-video-tool/deploy/backup-game-video-data.sh)。
2. 记录备份文件路径。
3. 检查归档文件是否生成。
4. 检查日志中是否记录成功。

R2 中风险部署优先执行范围备份：

1. 明确受影响的数据库、用户目录、文件子目录或配置文件。
2. 只备份相关范围，避免每次生成 7G+ 全量包。
3. 若无法明确范围、涉及多人并发写入或影响费用/账号安全，升级为 R3。

R0/R1 默认不执行生产数据全量备份，但仍必须确认发布包不会覆盖 `game-video-data`。

### 7.4 备份验真动作

至少检查：

1. 备份文件真实存在。
2. 文件大小合理，不是 0。
3. 备份时间与当前操作吻合。
4. 备份日志可追溯。

## 8. 验收失败 SOP

若上线后验收失败：

1. 立刻停止继续操作。
2. 记录失败点：
   - 健康检查失败
   - 首页失败
   - 登录失败
   - 关键功能失败
3. 判断是否能在短时间内修复。
4. 若不能，执行代码回滚。
5. 回滚后重新验收。

## 10. 本项目当前标准

结合现状，本项目当前应遵守以下硬标准：

1. 测试未通过不部署。
2. 推送前必须询问用户。
3. 部署前必须询问用户。
4. 部署时不覆盖 `/home/deploy/game-video-data`。
5. 高风险改动优先执行服务器数据备份。
6. 出现严重异常优先回滚代码。

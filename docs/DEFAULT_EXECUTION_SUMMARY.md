# 默认执行摘要

以后用户只要说一句“按默认流程来”，默认按本文件执行。

## 1. 默认动作

1. 先读代码，判断影响范围。
2. 直接实施必要改动，不停留在空泛方案。
3. 按改动类型执行对应测试。
4. 先汇报测试结果。
5. 测试通过后自动本地提交。
6. 推送前必须征求用户确认。
7. 部署前必须征求用户确认。
8. 上线后必须做健康检查和关键路径验收。

## 2. 强制门禁

以下任一条件不满足，都不得上线：

1. 测试未通过。
2. 没有清晰回滚点。
3. 不清楚是否会影响数据目录。
4. 用户未确认部署。

R0 文档包例外：不部署生产、不重启服务、不做生产 DB 备份；完成本地检查和本地提交即可。

## 2.1 硬性规则护栏

每次开工前必须先按 [项目硬性规则护栏](/Users/jinyu/Documents/game-video-tool/docs/HARD_RULES.md) 自检，尤其确认：

1. 没有用户明确允许，不部署生产、不重启服务、不写生产数据库。
2. DB 定向备份不得包含用户媒体目录。
3. 不做模型 fallback；上游失败必须明确失败。
4. 用户可见分析、脚本、提示词和反推结果必须直接用简体中文生成。
5. 遇到性能问题先观测分类，再优化，不凭感觉乱补。

## 3. 自动提交规则

默认开启自动提交，但必须满足：

1. 改动已完成。
2. 测试通过。
3. 提交内容干净、主题单一。

## 4. 推送规则

默认不自动推送。推送前必须说明：

1. 当前分支。
2. 最新提交号。
3. 测试结论。

## 5. 部署规则

默认不自动部署。部署前必须确认：

1. 只覆盖代码，不覆盖数据目录。
2. 备份策略已满足。
3. 健康检查命令已准备。
4. 首页验收命令已准备。

## 6. 备份规则

本项目默认同时依赖三层备份：

1. Git 提交和 GitHub：代码备份。
2. 本地自动数据库备份：运行数据备份。
3. 服务器数据归档：生产数据备份。

执行时不再默认每次都做生产数据全量备份，必须先做备份风险分级：

1. R0 文档、提示词、纯说明改动：不需要生产数据备份，只确认工作区和提交点。
2. R1 纯前端、样式、只读接口、无数据写入改动：需要代码备份和回滚点，不做全量数据备份。
3. R2 影响单个功能的数据写入、文件输出、费用记录、任务状态：备份相关数据库或相关目录；如无法精确定位，再升级到 R3。
4. R3 认证、权限、账号、数据库结构、数据迁移、同步、部署脚本、数据目录路径、大版本上线：上线前必须做生产数据全量备份。

任何级别都必须确认部署包不会覆盖 `game-video-data`。如果风险分级无法判断，按更高一级执行。

## 7. 汇报规则

每次阶段性反馈默认包含：

1. 改了什么。
2. 测了什么。
3. 结果如何。
4. 是否需要你确认下一步。

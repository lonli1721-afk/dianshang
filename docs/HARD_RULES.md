# 项目硬性规则护栏

本文件是所有窗口、分支、agent 和工程师的开工前自检清单。任何规则冲突时，以本文件和用户最新明确指令为准。

## 1. 生产权限

1. 未经用户明确说出“允许上线”“允许部署”“允许执行生产修复”“允许写线上数据”，不得部署生产、重启服务、写生产数据库或执行 repair。
2. “继续”“按你的建议继续”“好的”不等于生产授权，只能继续本地实现、测试、审查和准备上线材料。
3. 紧急问题也必须先只读定位，说明拟执行动作、备份点和回滚方式，再等待明确授权。

## 2. 上线与回滚

1. 上线前必须测试、代码审查、提交点清晰、包干净、备份策略明确。
2. 每次只做一个主题，独立提交、独立测试、独立上线，便于回滚。
3. 上线后必须检查服务 active、`/health 200`、首页资源、日志、provider queue、task audit。
4. 出现健康检查失败、页面空白、集中 500、stale processing 增长或合法请求明显误伤，立即回滚代码。

## 3. 备份与数据

1. 备份必须按风险分级执行，不得默认每次全量备份生产媒体。
2. DB 定向备份只包含 `auth.db`、`game_video.db`、`users/*/database.db`、`cloud-dbs`，禁止误把用户媒体目录打入 DB 备份。
3. 不覆盖 `game-video-data`，不删除用户生成文件。
4. 媒体清理必须 dry-run -> 人工审核 -> 明确批准 -> execute。
5. 已删除项目的费用记录永久保留，不能因项目删除而丢账。

## 4. 模型与语言

1. 不做模型 fallback；上游繁忙、限流、权限、欠费或参数错误必须明确失败。
2. 所有用户可见分析、脚本、提示词、反推结果必须直接用简体中文生成。
3. 禁止“英文提示词生成后再翻译成中文”的路径。
4. 如果模型仍返回英文，必须拦截并提示用户重新生成。

## 5. 架构与分支

1. `prod-ops` 只做线上热修、稳定治理和最小上线包。
2. 新功能先在 `feature-lab` 本地实现，稳定后交由 `architecture` 审查，再由 `prod-ops` 做最小稳定上线包。
3. 子组件只做展示和事件转发；业务副作用留在 hook 或父级 orchestrator。
4. 后端 router 只做请求校验和响应转换；provider 分支逻辑应逐步下沉到 service/registry。

## 6. 性能原则

1. 多人并发、响应迅速、可定位、可回滚优先于新增功能。
2. 遇到“慢”，先用观测区分项目加载、媒体预览、任务轮询、模型请求，禁止凭感觉乱补。
3. 模型请求慢优先看 provider queue、key 池、429/503；项目加载慢再看 DB 查询、场景 JSON、前端渲染和缓存。

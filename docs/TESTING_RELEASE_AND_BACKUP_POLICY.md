# 测试、上线与备份策略

本文档定义本项目的测试门禁、上线门禁、回滚标准和备份要求。目标不是“看起来差不多能用”，而是尽量把每次发布都做成可验证、可回滚、可追踪的操作。

## 1. 发布总门禁

要进入线上部署，必须同时满足以下条件：

1. 改动目标已经明确。
2. 改动已经完成。
3. 与改动相关的测试全部通过。
4. 测试结果已经告知用户。
5. 已有可识别的 Git 提交点。
6. 已有回滚方案。
7. 用户明确同意部署。

其中第 3 条为硬门禁，不得跳过。

## 2. 改动分类与测试要求

### 2.1 前端改动

最少测试集：

1. 依赖完整性确认。
2. `react-ui` 构建成功。
3. 如可执行，运行页面并验证关键交互未报错。
4. 检查被改页面的核心路径是否可打开、可提交、可响应。

推荐命令：

```bash
cd react-ui
npm run build
```

如本机具备 lint 运行条件，补充：

```bash
cd react-ui
npm run lint
```

重点观察：

1. 构建是否失败。
2. 控制台是否存在明显报错。
3. 页面是否出现空白、报错、关键按钮不可用。

### 2.2 后端改动

最少测试集：

1. 后端依赖可用。
2. 服务可以正常启动。
3. `/health` 返回 200。
4. 与改动相关的接口可成功调用。
5. 认证、权限、同步、文件处理等关键链路在改动范围内必须回归验证。

推荐命令：

```bash
.venv/bin/python server/main.py --host 127.0.0.1 --port 57991
curl http://127.0.0.1:57991/health
```

重点观察：

1. 是否启动即崩溃。
2. 是否出现导入错误、权限错误、路径错误。
3. 接口返回是否符合预期。
4. 是否引入明显性能回退。

### 2.3 部署相关改动

最少测试集：

1. 本地构建通过。
2. 部署命令、打包命令、脚本参数语义正确。
3. 健康检查命令可用。
4. 首页和静态资源能正确访问。

重点观察：

1. 路径是否正确。
2. 服务重启后是否能启动。
3. 验收命令是否真的能反映线上状态。

### 2.4 认证、权限、同步、备份类改动

该类改动属于高风险改动，必须额外验证：

1. 登录流程。
2. 角色权限边界。
3. 关键同步接口。
4. 自动备份逻辑是否仍按预期运行。
5. 错误配置下是否会导致递归、死循环、异常请求放大或资源打满。

## 3. 本项目当前可用的测试基线

根据当前仓库结构，本项目至少应使用以下测试基线：

### 3.1 前端基线

1. `react-ui/package.json` 中存在 `build` 和 `lint`。
2. 若环境支持 Node/npm，则优先执行：

```bash
cd react-ui
npm run build
npm run lint
```

### 3.2 后端基线

1. 使用项目虚拟环境 `.venv`。
2. 启动命令：

```bash
env USER_DATA_DIR=/path/to/local-data JWT_SECRET=local-dev-secret AUTH_ENABLED=true .venv/bin/python server/main.py --host 127.0.0.1 --port 57991
```

3. 健康检查：

```bash
curl -sS -o /dev/null -w "code:%{http_code} ttfb:%{time_starttransfer} total:%{time_total}\n" http://127.0.0.1:57991/health
```

4. 首页检查：

```bash
curl -sS http://127.0.0.1:57991/
```

## 4. 测试执行顺序

推荐顺序如下：

1. 静态检查或构建检查。
2. 服务启动检查。
3. 健康检查。
4. 核心功能验证。
5. 回归验证。
6. 汇总结果。

原因：

1. 构建失败通常最便宜、最早发现。
2. 服务起不来时，后续接口验证没有意义。
3. 健康检查通过后再做功能验证更高效。

## 5. 测试记录要求

每次提交前至少记录以下内容：

1. 改动范围。
2. 执行了哪些测试。
3. 哪些测试通过。
4. 哪些测试未执行，以及原因。
5. 是否允许进入下一步。

标准汇报模板：

1. 本次改动：
2. 已执行测试：
3. 结果：
4. 未执行项：
5. 风险说明：
6. 是否建议推送：
7. 是否建议部署：

## 6. 上线前检查单

部署前必须逐项确认：

1. 当前代码已提交到 Git。
2. 部署目标版本明确。
3. 生产数据目录不在覆盖范围内。
4. 已完成备份风险分级，并按分级执行对应备份策略。
5. 部署命令正确。
6. 回滚包或回滚提交存在。
7. 健康检查命令可执行。
8. 首页访问验收命令可执行。

## 7. 本项目的备份策略

本项目备份分三层。

### 7.1 代码备份

要求：

1. 每次稳定改动至少有一个 Git 提交点。
2. 重要版本应有可识别标签或发布包。
3. 推送到 GitHub 后，远程仓库构成代码异地备份。

### 7.2 本地自动备份

当前项目已有自动备份逻辑，位于 [`server/cloud_sync.py`](/Users/jinyu/Documents/game-video-tool/server/cloud_sync.py)。

已知行为：

1. 自动备份数据库。
2. 默认写入 `USER_DATA_DIR/backups/auto/`。
3. 默认按数量保留历史备份目录。

使用要求：

1. 本地调试时不要关闭该能力的相关验证意识。
2. 涉及数据库行为改动时，应注意备份目录是否仍然生成。

### 7.3 服务器数据备份

服务器已有脚本 [`deploy/backup-game-video-data.sh`](/Users/jinyu/Documents/game-video-tool/deploy/backup-game-video-data.sh)。

已知行为：

1. 对 `GAME_VIDEO_DATA_DIR` 做快照复制。
2. 对其中的 SQLite 数据库执行 `.backup`。
3. 产出 `tar.gz` 归档文件。
4. 按保留天数清理旧备份。

上线前建议：

1. 只有 R3 级改动默认执行服务器数据全量备份。
2. R2 级改动优先备份相关数据库或相关目录；无法明确影响范围时升级为 R3。
3. 发布后若出现严重问题，优先代码回滚；不要直接操作生产数据，除非已经确认恢复方案。

### 7.4 上线备份风险分级

上线前必须先定级，再备份。不得把“全量生产数据备份”作为所有发布的默认动作。

| 级别 | 适用改动 | 必需备份 | 是否全量备份生产数据 |
| --- | --- | --- | --- |
| R0 | 文档、提示词、交接材料、注释说明，不影响运行代码 | 代码提交或文档版本留存 | 否 |
| R1 | 纯前端展示、样式、只读接口、无数据写入的修复 | 代码提交、发布包、回滚点 | 否 |
| R2 | 单个功能的数据写入、文件输出、费用记录、任务状态、模型结果缓存 | 代码备份 + 相关数据库或相关目录备份 | 视影响范围决定 |
| R3 | 登录、权限、账号、数据库结构、数据迁移、同步、备份脚本、部署脚本、数据目录路径、大版本上线 | 代码备份 + 生产数据全量备份 | 是 |

定级规则：

1. 无法判断影响范围时，按更高一级执行。
2. 任何级别都必须确认部署包不会覆盖 `game-video-data`。
3. R1 及以下禁止为了“保险”无脑打全量数据包，除非用户明确要求或线上状态异常。
4. R2 如果只影响某个用户、某个数据库或某类生成文件，应优先做范围备份，避免占满磁盘。
5. R3 全量备份前必须检查磁盘空间；备份后必须验真文件存在、非空、可读。

## 8. 自动备份的理解与执行标准

“懂得自动备份”在本项目中意味着：

1. 代码层面依靠 Git 提交和远程仓库进行版本留存。
2. 本地运行层面依靠应用内数据库自动备份。
3. 生产环境层面依靠服务器脚本归档数据目录。
4. 上线前清楚区分“代码覆盖”和“数据保护”。

换句话说：

1. Git 不是数据备份。
2. 数据目录备份不能替代代码版本控制。
3. 打包发布包不能替代数据库归档。

## 9. 回滚策略

本项目默认回滚原则：

1. 优先回滚代码。
2. 谨慎处理数据。
3. 只有在数据损坏或明确需要恢复时，才考虑使用数据备份。

推荐顺序：

1. 回到上一个稳定 Git 提交或历史发布包。
2. 重新部署代码。
3. 执行健康检查和首页验收。
4. 若仍异常，再判断是否需要数据恢复。

## 10. 上线后验收

最低验收标准：

1. `GET /health` 返回 200。
2. 首页可访问。
3. 前端静态资源正确加载。
4. 与本次改动相关的关键功能至少验证一次。
5. 日志中无明显启动失败或持续报错。

典型验收命令：

```bash
curl -sS -o /dev/null -w "code:%{http_code} ttfb:%{time_starttransfer} total:%{time_total}\n" http://线上地址/health
curl -sS http://线上地址/ | grep -i "assets/index-"
```

## 11. 不允许上线的情况

出现以下任一情况，不得上线：

1. 构建失败。
2. 服务无法启动。
3. 健康检查失败。
4. 关键接口失败。
5. 改动涉及高风险链路但未完成回归验证。
6. 无法确认回滚方案。
7. 用户未确认部署。

## 12. 推荐发布节奏

推荐节奏如下：

1. 本地修改。
2. 本地测试。
3. 本地提交。
4. 用户确认后推送。
5. 用户确认后部署。
6. 上线验收。
7. 如有必要，补充更新交接文档。

## 13. 文档维护要求

当项目新增正式测试框架、CI/CD、自动化发布流程或新的备份脚本时，必须同步更新本文档。

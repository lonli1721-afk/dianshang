# 变更检查清单

本文档是本项目的执行级检查单。目标是把“测试过了”“可以上线了”从主观判断变成逐项核对。

使用方式：

1. 先判断本次改动属于哪一类。
2. 按对应清单逐项核对。
3. 未完成项必须明确写出原因。
4. 任何阻塞项存在时，不得上线。

## 1. 通用检查清单

适用于所有改动。

### 1.1 需求澄清

- [ ] 已明确本次改动目标。
- [ ] 已明确影响范围。
- [ ] 已确认是否涉及前端。
- [ ] 已确认是否涉及后端。
- [ ] 已确认是否涉及部署。
- [ ] 已确认是否涉及认证、权限、同步、备份、数据。

### 1.2 代码卫生

- [ ] 改动范围控制在必要文件内。
- [ ] 未引入明显的无关重构。
- [ ] 未提交密钥、密码、token、JWT_SECRET。
- [ ] 未提交数据库、缓存、用户数据、日志、构建产物。
- [ ] `.gitignore` 仍然有效覆盖本地产物。

### 1.3 提交前状态

- [ ] 已确认测试范围。
- [ ] 已执行相关测试。
- [ ] 已记录未执行项与原因。
- [ ] 已确认当前改动可形成独立提交。
- [ ] 已确认有回滚点。
- [ ] 若涉及生产动作，已确认用户明确说过“允许上线/允许部署/允许执行生产修复/允许写线上数据”，不能把“继续”当作授权。

## 2. 前端改动清单

适用于 `react-ui` 相关改动。

### 2.1 编译与静态检查

- [ ] 前端依赖可用。
- [ ] `npm run build` 通过。
- [ ] 若环境可执行，`npm run lint` 通过。
- [ ] 无新增构建警告导致的明显风险。

### 2.2 页面行为

- [ ] 被改页面可以正常打开。
- [ ] 页面无白屏。
- [ ] 页面无显式报错提示。
- [ ] 关键按钮可点击。
- [ ] 关键按钮状态切换符合预期。
- [ ] 表单输入、提交、提示文案符合预期。
- [ ] 条件渲染未出现 `0`、`undefined`、空白块等异常显示。
- [ ] 文本在桌面和窄宽度下未溢出。

### 2.3 接口联动

- [ ] 页面调用的关键接口仍然成功。
- [ ] 失败提示能正确展示。
- [ ] 登录态/Token 相关流程未被破坏。
- [ ] 权限受限页面仍能正确处理 401/403。
- [ ] 前端上线时已保留上一版 hashed chunk，已验证当前主资源和上一版关键 chunk 均返回 200。

### 2.4 阻塞上线条件

若出现以下任一情况，前端改动禁止上线：

- [ ] 构建失败。
- [ ] 关键页面打不开。
- [ ] 页面白屏。
- [ ] 核心按钮不可用。
- [ ] 关键交互报错。
- [ ] 当前 `index.html` 引用的静态资源 404，或上线后新增动态 chunk 404 / `Failed to fetch dynamically imported module`。

## 3. 后端改动清单

适用于 `server` 相关改动。

### 3.1 启动与依赖

- [ ] 后端依赖可用。
- [ ] 服务可正常启动。
- [ ] 启动日志无致命异常。
- [ ] `/health` 返回 200。

### 3.2 接口验证

- [ ] 与改动相关的接口至少验证一遍。
- [ ] 成功路径返回符合预期。
- [ ] 错误路径返回符合预期。
- [ ] 认证失败时返回符合预期。
- [ ] 权限不足时返回符合预期。

### 3.3 文件与路径

- [ ] 本地数据目录行为符合预期。
- [ ] 不会误写到受限或错误路径。
- [ ] 不会覆盖不该覆盖的文件。
- [ ] 若涉及静态资源挂载，首页仍能访问。

### 3.4 性能与稳定性

- [ ] 没有引入明显死循环或递归调用风险。
- [ ] 没有引入显著阻塞事件循环的同步重活。
- [ ] 高风险逻辑已检查是否可能导致 CPU 打满。
- [ ] 网络调用超时、失败、重试路径合理。

### 3.5 阻塞上线条件

若出现以下任一情况，后端改动禁止上线：

- [ ] 服务无法启动。
- [ ] `/health` 失败。
- [ ] 关键接口失败。
- [ ] 登录/鉴权异常。
- [ ] 明显资源打满风险未消除。
- [ ] 用户尚未明确授权生产部署、服务重启或线上数据写入。

## 4. 认证与权限改动清单

适用于登录、账号、角色、权限边界相关改动。

### 4.1 登录链路

- [ ] 正常登录成功。
- [ ] 错误密码登录失败。
- [ ] 未登录访问受保护接口被拒绝。
- [ ] Token 过期或无效时行为正确。
- [ ] 不存在回连自身导致的递归登录风险。

### 4.2 角色权限

- [ ] 管理员权限仍可用。
- [ ] 普通用户不能访问管理员接口。
- [ ] 用户自己的设置仍然可读写。
- [ ] 系统级设置边界没有被放开。

### 4.3 用户数据隔离

- [ ] 用户数据库路径切换符合预期。
- [ ] 用户文件目录切换符合预期。
- [ ] 不会串用其他用户的数据。

### 4.4 阻塞上线条件

- [ ] 普通用户能访问管理员接口。
- [ ] 登录失败但返回成功。
- [ ] 用户数据隔离失效。
- [ ] 存在递归登录或鉴权死循环风险。

## 5. 同步与备份改动清单

适用于云同步、本地备份、服务器备份相关改动。

### 5.1 同步链路

- [ ] 文件同步队列行为正常。
- [ ] 数据库推送逻辑行为正常。
- [ ] 状态面板信息未失真。
- [ ] 失败时错误信息可观察。

### 5.2 本地自动备份

- [ ] 自动备份目录仍然生成。
- [ ] 备份周期逻辑未被破坏。
- [ ] 备份保留策略未被破坏。
- [ ] 涉及数据库改动时，备份文件可被创建。

### 5.3 服务器备份

- [ ] 已完成备份风险分级：R0 / R1 / R2 / R3。
- [ ] 已确认服务器备份脚本路径。
- [ ] 已确认备份目录有可写空间。
- [ ] 已确认保留策略合理。
- [ ] R0/R1 已确认不需要生产数据全量备份。
- [ ] R2 已确认范围备份对象，或已升级为 R3。
- [ ] R3 上线前已执行或计划执行一次生产数据全量备份。

### 5.4 阻塞上线条件

- [ ] 同步失败且无错误可观测信息。
- [ ] 自动备份逻辑失效。
- [ ] 未完成备份风险分级。
- [ ] R2/R3 改动没有对应备份方案。

## 6. 部署改动清单

适用于发布流程、部署脚本、服务脚本、Docker、Nginx 等改动。

### 6.1 打包与发布

- [ ] 构建命令正确。
- [ ] 打包命令正确。
- [ ] 发布包内容范围正确。
- [ ] 不会把数据目录打进代码包。

### 6.2 服务运行

- [ ] 服务启动命令正确。
- [ ] 服务重启命令正确。
- [ ] 端口与代理配置正确。
- [ ] 健康检查命令正确。

### 6.3 上线验收

- [ ] `/health` 可访问且返回 200。
- [ ] 首页可访问。
- [ ] 前端资源 hash 正常加载。
- [ ] 前端 `dist` 替换使用了保留旧 chunk 的流程，未直接清空旧 `assets`。
- [ ] 本次改动的关键路径可用。

### 6.4 阻塞上线条件

- [ ] 发布包内容错误。
- [ ] 服务脚本错误。
- [ ] Nginx/代理配置错误。
- [ ] 上线后首页或健康检查失败。

## 7. 文档改动清单

适用于 README、交接文档、规范文档改动。

### 7.1 文档质量

- [ ] 说明与项目现状一致。
- [ ] 路径、命令、端口、地址没有写错。
- [ ] 若引用现有脚本或文件，路径可以打开。
- [ ] 不包含敏感信息。

### 7.2 阻塞上线条件

文档改动通常不阻塞业务上线，但若文档用于直接指导发布，则以下情况视为阻塞：

- [ ] 发布步骤错误。
- [ ] 备份步骤错误。
- [ ] 回滚步骤错误。

## 8. 最终放行清单

只有以下项全部满足，才允许进入部署确认：

- [ ] 本次改动已完成。
- [ ] 相关检查单已通过。
- [ ] 测试结果已汇报给用户。
- [ ] 已形成本地 Git 提交。
- [ ] 已有回滚点。
- [ ] 已确认数据目录不会被覆盖。
- [ ] 用户已准备决定是否推送/部署。

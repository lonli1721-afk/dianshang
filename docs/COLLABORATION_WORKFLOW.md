# 协作流程约定

本文档定义本项目与 Codex/AI 协作时的默认工作方式。除非用户明确覆盖，所有需求默认按本文档执行。

## 1. 总体原则

1. 默认允许自动修改代码。
2. 默认允许自动提交到本地 Git。
3. 自动推送到远程仓库前，必须得到一次用户确认。
4. 自动部署到服务器前，必须得到一次用户确认。
5. 测试未通过，禁止上线。
6. 每次上线前必须先确认备份策略已经满足要求。
7. 任何操作都不得覆盖或删除生产数据目录。

## 2. 默认执行顺序

每次需求默认按以下顺序推进：

1. 阅读代码，确认影响范围。
2. 判断改动类型：
   - 前端
   - 后端
   - 配置
   - 部署
   - 数据/权限
   - 混合改动
3. 实施代码修改。
4. 执行与改动范围对应的测试。
5. 汇总测试结果。
6. 测试通过后，自动创建本地 Git 提交。
7. 询问用户是否允许推送。
8. 推送完成后，询问用户是否允许部署。
9. 部署完成后执行上线验收。
10. 返回最终结果、风险说明和回滚点。

## 3. 自动提交规则

默认开启自动提交。规则如下：

1. 仅在测试通过后提交。
2. 若改动仍处于中间状态，不提交半成品，除非用户要求保存检查点。
3. 提交信息必须简洁、可回溯、与实际改动一致。
4. 一次提交只覆盖一个相对清晰的改动主题。
5. 不把本地运行产物、数据库、缓存、密钥文件纳入提交。

## 4. 提交信息规则

推荐使用英文祈使句，遵循“动作 + 目标”风格。

示例：

- `fix login recursion on cloud auth`
- `improve sync status panel styling`
- `add game key settings for users`
- `document release and backup workflow`

不推荐：

- `update`
- `fix bugs`
- `change files`

## 5. 对用户的汇报格式

每次完成一个阶段，默认按以下结构汇报：

1. 改了什么
2. 为什么这样改
3. 测了什么
4. 当前结果
5. 是否需要用户确认下一步

如果是较大改动，补充说明：

- 影响模块
- 风险点
- 是否涉及部署
- 是否涉及数据兼容

## 6. 自动推送规则

自动推送默认关闭，必须用户确认后执行。

执行条件：

1. 测试通过。
2. 本地工作区状态清晰。
3. 已完成本地提交。
4. 用户明确同意推送。

推送前应告知用户：

- 当前分支
- 最新提交信息
- 推送目标远程仓库

## 7. 自动部署规则

自动部署默认关闭，必须用户确认后执行。

重要补充：

1. “继续”“按你的建议继续”“好的继续”只代表继续分析、本地实现、本地测试、本地提交或准备上线材料。
2. 这些表达不等于允许上线、不等于允许重启服务、不等于允许写生产数据库。
3. 只有用户明确说出“允许上线”“允许部署”“允许执行生产修复”“允许写线上数据”之一，才允许执行生产部署、服务重启、线上数据写入或 repair 脚本。
4. 若任务紧急，也必须先完成只读定位，说明拟执行动作、备份点、回滚方式，并等待明确授权。

执行条件：

1. 所有测试通过。
2. 代码已完成本地提交。
3. 如需，代码已推送到远程仓库。
4. 备份策略已执行或已确认满足要求。
5. 用户明确同意部署。

部署前必须确认：

1. 只覆盖代码，不覆盖数据目录。
2. 本次部署对应的代码版本可识别。
3. 已有回滚点。
4. 健康检查和首页验收命令准备完毕。

## 8. 测试失败时的规则

如果测试失败，默认执行：

1. 停止推送。
2. 停止部署。
3. 先定位失败原因。
4. 修复后重新测试。
5. 只有测试重新通过，才允许进入提交/推送/部署环节。

不得做的事情：

1. 不得在已知失败状态下强行上线。
2. 不得隐瞒失败结果。
3. 不得跳过关键验证步骤。

## 9. 备份规则

本项目必须同时理解三类备份：

1. 代码备份：Git 提交、Git 远程仓库、发布包。
2. 本地数据备份：应用运行目录下的自动数据库备份。
3. 服务器数据备份：生产数据目录的归档备份。

执行原则：

1. 修改代码后，Git 提交是最基本的代码备份。
2. 上线前，必须确认生产数据目录不会被覆盖。
3. 上线前必须先做备份风险分级，不得默认每次全量备份生产数据。
4. R0/R1 只做代码回滚点和发布包，不做生产数据全量备份。
5. R2 优先做范围备份；影响范围无法判断时升级为 R3。
6. R3 才默认执行生产数据全量备份。

风险分级：

1. R0：文档、提示词、交接材料、注释说明。
2. R1：纯前端、样式、只读接口、无数据写入。
3. R2：单功能写入、文件输出、费用记录、任务状态、模型结果缓存。
4. R3：认证、权限、账号、数据库结构、数据迁移、同步、备份/部署脚本、数据目录路径、大版本上线。

## 10. 备份相关的项目现状

当前项目已经存在以下备份能力：

1. 后端 `server/cloud_sync.py` 会对本地数据库做自动备份，默认写入 `USER_DATA_DIR/backups/auto/`。
2. 服务器侧已有脚本 [`deploy/backup-game-video-data.sh`](/Users/jinyu/Documents/game-video-tool/deploy/backup-game-video-data.sh)，用于打包 `game-video-data` 数据目录并按保留天数清理旧备份。
3. 交接文档明确要求：部署时只回滚代码，不动数据目录。

## 11. 分支建议

默认分支策略如下：

1. `main`：可部署、相对稳定的版本。
2. `feature/*`：新功能开发。
3. `fix/*`：缺陷修复。
4. `docs/*`：文档类改动。

推荐流程：

1. 在功能分支开发。
2. 测试通过后提交。
3. 推送分支。
4. 合并到 `main`。
5. 用 `main` 对应版本部署。

## 12. 默认协作口令

以后用户若使用以下表达，按固定流程执行：

- “按默认流程改这个需求”
- “改完自动提交”
- “先测试，测过再推”
- “按规范改、存档、上传”

正式执行前，先阅读并遵守 [项目硬性规则护栏](/Users/jinyu/Documents/game-video-tool/docs/HARD_RULES.md)。该文件是多窗口、多 agent 并行协作时的统一硬约束。

其含义统一为：

1. 自动改代码。
2. 自动测试。
3. 测试结果先汇报。
4. 自动提交。
5. 推送前确认。
6. 部署前确认。

以下表达必须按“仅准备、不生产执行”理解：

- “继续”
- “接着做”
- “按照你的建议继续”
- “好的，继续”

其含义统一为：

1. 可以继续本地分析、代码修改、测试、提交。
2. 可以准备部署包和上线说明。
3. 不可以部署、不可以重启服务、不可以写线上数据库、不可以执行 repair。
4. 必须在生产动作前再次等待用户明确授权。

## 13. 禁止事项

1. 不得把密钥、密码、JWT_SECRET、云端 token 提交进 Git。
2. 不得把数据库、用户文件、缓存目录提交进 Git。
3. 不得在未测试通过时部署到线上。
4. 不得覆盖生产数据目录。
5. 不得在未说明风险的情况下执行高风险操作。

## 14. 文档维护规则

如果默认流程发生变化，必须同步更新本文档，并在提交信息中明确标注文档更新。

## 15. Agent 协作规则

默认可以使用多个 agent 提高审查质量，但必须控制写入边界。

1. 正式执行前可开 2-3 个只读 agent 做架构审查、测试审查、风险排查。
2. 默认 agent 只读，不修改文件、不部署、不提交。
3. 多个 agent 不得同时修改同一文件。
4. 只有用户明确要求实施代码时，才允许 worker agent 做代码修改。
5. 主线程负责整合结论、最终取舍、提交、打包和上线。
6. agent 输出不能直接作为上线结论，必须由主线程复核。
